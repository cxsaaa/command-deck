## 1. 产品概述

### 1.1 产品定位

本产品是一款面向开发者的 Mac 桌面端命令速查应用。

用户可以通过选择平台、搜索关键词、浏览分类等方式，快速找到常用命令，查看简短说明，并一键复制命令到剪贴板。

产品重点不是“命令管理后台”，而是“高频命令查找与复制工具”。

### 1.2 一句话描述

一个让开发者按平台快速查找、理解并复制常用命令的 Mac 桌面端应用。

### 1.3 核心价值

- 快速找到命令
- 快速理解命令用途
- 快速复制命令
- 按平台组织命令
- 降低记忆成本
- 避免重复搜索和翻笔记

---

## 2. 背景与问题

### 2.1 用户现状

开发者在日常工作中会频繁使用各种命令行工具，例如：

- Docker
- Claude Code
- CLI
- OpenCode
- Git
- Homebrew
- Kubernetes
- Node.js
- Python
- Shell

这些命令通常分散在：

- 浏览器收藏夹
- Notion / Obsidian / Markdown 文档
- 项目 README
- 终端历史记录
- ChatGPT 对话记录
- 个人代码片段库

### 2.2 当前痛点

1. 命令分散，查找成本高  
   用户需要在多个地方搜索命令，尤其是不同平台命令混在一起时，查找效率低。

2. 命令容易忘记细节  
   例如 Docker、Git、Kubernetes 等命令参数较多，用户经常记得大概用途，但忘记准确写法。

3. 通用笔记工具不适合命令速查  
   Notion、Obsidian、Markdown 文档适合记录，但不适合高频搜索、筛选、一键复制。

4. CRUD 并非高频动作  
   用户更多是在使用已有命令，而不是频繁新增、编辑、删除命令。

5. 平台维度很重要  
   用户的自然使用习惯通常是：先想到平台，再找该平台下的命令。

---

## 3. 产品目标

### 3.1 第一阶段目标

构建一个轻量、清晰、快速的 Mac 命令速查工具。

用户可以：

- 选择一个平台
- 查看该平台下的命令列表
- 搜索命令、描述、标签、参数
- 一键复制命令
- 收藏常用命令
- 查看最近使用命令
- 低频维护命令数据

### 3.2 非目标

第一版暂不重点解决：

- 团队协作
- 云端同步
- 账号体系
- 命令市场
- 远程执行命令
- 复杂 AI 生成命令
- 多人权限管理
- Web 端版本

## 5. 核心使用场景

### 5.1 场景一：按平台查找命令

用户想查看 Docker 相关命令。

操作路径：

1. 打开应用
2. 左侧选择 Docker 平台
3. 右侧显示 Docker 命令列表
4. 浏览分类或常用命令
5. 点击复制

### 5.2 场景二：搜索命令

用户记得命令和“logs”相关，但不记得完整写法。

操作路径：

1. 打开应用
2. 使用快捷键 `⌘K` 聚焦搜索框
3. 输入 `logs`
4. 查看搜索结果
5. 找到 `docker logs -f <container_name>`
6. 点击复制或按 Enter 复制当前选中命令

### 5.3 场景三：复制常用命令

用户经常使用某些命令，例如查看容器、进入容器、查看日志。

操作路径：

1. 打开应用
2. 进入收藏或最近使用
3. 找到目标命令
4. 点击复制

### 5.4 场景四：维护命令

用户发现某条命令需要更新。

操作路径：

1. 找到命令卡片
2. 点击更多菜单 `···`
3. 选择编辑
4. 修改命令内容、描述、平台、分类或标签
5. 保存

## 6. 产品原则

### 6.1 搜索优先

搜索是核心能力，不是辅助功能。

搜索框应始终位于主界面顶部，并支持快捷键快速聚焦。

### 6.2 平台优先

平台是主要组织方式。

用户默认通过平台进入命令列表，例如 Docker、Claude Code、CLI、OpenCode。

### 6.3 命令展示优先

主界面重点展示命令内容、描述和复制操作。

新增、编辑、删除等维护操作应弱化，不应干扰主使用路径。

### 6.4 复制优先

复制是用户最终动作。

每张命令卡片上最显眼的操作应该是“复制”。

### 6.5 简洁清晰

界面应避免表格化、后台化、复杂表单化。

优先使用卡片列表、轻量分组、清晰层级。

主界面是用户最高频访问页面，主要用于选择平台、搜索命令、浏览命令和复制命令。

推荐采用两栏布局：

```text
┌──────────────────────────────────────────────────────┐
│ 应用标题 / 搜索框 / 新建入口 / 设置入口               │
├───────────────┬──────────────────────────────────────┤
│ 左侧导航栏     │ 右侧命令展示区                       │
│               │                                      │
│ 全部           │ 当前平台标题                          │
│ 收藏           │ 分类筛选                              │
│ 最近使用       │ 命令卡片列表                          │
│               │                                      │
│ 平台列表       │                                      │
│ Docker         │                                      │
│ Claude Code    │                                      │
│ CLI            │                                      │
│ OpenCode       │                                      │
└───────────────┴──────────────────────────────────────┘
```

### 8.1.3 左侧导航栏

左侧导航栏包含：

- 全部命令

- 最近使用
- 平台列表

每个平台显示：

- 平台名称
- 命令数量


## 8.2 命令卡片

### 8.2.1 功能说明

命令卡片是命令的主要展示单元。

卡片应突出命令标题、命令内容、描述和复制按钮。

### 8.2.2 卡片基础信息

每张卡片默认展示：

- 命令标题
- 命令内容
- 一句话描述
- 标签或分类
- 复制按钮
- 收藏按钮
- 更多操作菜单

### 8.2.3 卡片示例

```text
┌─────────────────────────────────────────────┐
│ 查看 Docker 容器日志                    ☆ ··· │
│ docker logs -f <container_name>              │
│ 实时跟踪指定容器的日志输出。                  │
│ logs · debug · container              复制   │
└─────────────────────────────────────────────┘
```

### 8.2.4 主操作

卡片主操作只有一个：复制。

点击复制后：

- 命令内容写入剪贴板
- 显示轻提示：已复制
- 更新命令的 `usageCount`
- 更新命令的 `lastUsedAt`

### 8.2.5 低频操作

低频操作放在更多菜单中：

```text
···
- 编辑
- 复制
- 收藏 / 取消收藏
- 删除
```

删除需要二次确认。

## 8.3 搜索功能

### 8.3.1 功能定位

搜索是 P0 核心能力。

用户应该可以通过搜索快速定位命令。

### 8.3.2 搜索入口

搜索框固定在顶部。

占位文案：

```text
搜索命令、描述、标签、参数...
```

快捷键：

```text
⌘K：聚焦搜索框
Esc：清空搜索或退出搜索状态
↑ / ↓：切换搜索结果
Enter：复制当前选中命令
```

### 8.3.3 搜索范围

搜索应覆盖以下字段：

- 命令标题
- 命令正文
- 命令描述
- 平台名称
- 分类名称
- 标签
- 示例
- 参数说明

### 8.3.4 搜索范围控制

当用户已选择某个平台时，默认在当前平台内搜索。

搜索结果区应提供范围切换：

```text
当前范围：Docker    [搜索全部平台]
```

当用户选择“全部命令”时，默认全局搜索。

### 8.3.5 搜索结果展示

搜索时右侧展示搜索结果。

示例：

```text
搜索结果：logs
当前范围：Docker    [搜索全部平台]

┌─────────────────────────────────────────────┐
│ 查看容器日志                                  │
│ docker logs -f <container_name>               │
│ 实时查看 Docker 容器日志。                     │
│ Docker · logs · debug                  复制   │
└─────────────────────────────────────────────┘
```

### 8.3.6 搜索结果排序

搜索结果排序建议：

1. 标题精确匹配
2. 命令内容匹配
3. 标签匹配
4. 描述匹配
5. 收藏优先
6. 最近使用优先
7. 使用次数高优先

### 8.3.7 空结果状态

当没有搜索结果时，显示：

```text
没有找到相关命令

可以尝试：
- 使用更短关键词
- 搜索命令中的参数
- 切换到全部平台搜索

+ 添加这条命令
```

## 9. 排版与界面规范

### 9.1 整体风格

- 简洁
- 清晰
- 类 Mac 原生体验
- 避免后台管理感
- 避免信息过载
- 优先阅读效率和复制效率

### 9.2 主界面布局建议

推荐两栏：

- 左侧：导航和平台
- 右侧：搜索结果或命令卡片列表

不建议第一版使用复杂三栏，因为固定详情栏会挤占命令展示空间。

### 9.3 命令卡片视觉层级

卡片信息优先级：

1. 标题
2. 命令内容
3. 复制按钮
4. 描述
5. 标签
6. 收藏和更多操作

### 9.4 命令内容样式

命令内容应使用等宽字体。

示例：

```bash
docker logs -f <container_name>
```

命令区域应具有轻微背景区分，方便识别。

### 9.5 密度模式

后续可支持两种展示模式：

#### 舒适模式

适合阅读和浏览。

```text
查看容器日志

docker logs -f <container_name>

实时跟踪指定容器日志。
logs · debug                         复制
```

#### 紧凑模式

适合命令量较大时快速扫描。

```text
查看容器日志     docker logs -f <container_name>     复制
查看容器列表     docker ps                           复制
```

## 14. 交互流程

## 14.1 复制命令流程

```text
用户点击复制
→ 命令写入剪贴板
→ 显示“已复制”提示
→ usageCount + 1
→ 更新 lastUsedAt
→ 命令进入最近使用
```

```text
用户按 ⌘K
→ 搜索框聚焦
→ 输入关键词
→ 右侧展示结果
→ 使用 ↑ / ↓ 选择结果
```

```text
→ 显示“已保存”提示
```

```text
点击命令卡片更多菜单

→ 修改字段
→ 点击保存

```

## 16. 验收标准

## 18. 技术方案建议

### 18.1 方案 A：SwiftUI + SQLite

适合追求 Mac 原生体验。

优点：

- 原生体验好
- 性能好
- 系统集成能力强
- 适合后续菜单栏、全局快捷键、iCloud

缺点：

- 开发门槛较高
- 前期 UI 迭代速度可能较慢

### 18.2 方案 B：Tauri + React + SQLite

适合熟悉前端技术、希望快速迭代 UI 的团队。

优点：

- UI 开发效率高
- 应用体积相对 Electron 更小
- 跨平台潜力较好
- 适合快速验证 MVP

缺点：

- Mac 原生感不如 SwiftUI
- 需要处理前端与本地能力桥接

### 18.3 方案 C：Electron + React + SQLite

适合最快验证产品。

优点：

- 生态成熟
- 开发效率高
- 桌面端经验丰富

缺点：

- 应用体积较大
- 内存占用较高
- Mac 原生感一般

### 18.4 推荐

如果目标是快速做出 MVP，建议使用：

```text
Tauri + React + SQLite
```

如果目标是长期打造高质量 Mac 原生应用，建议使用：

```text
SwiftUI + SQLite
```

## 21. 风险与注意事项

### 21.1 搜索体验风险

如果搜索不够快、不够准，产品核心价值会下降。

应优先保证：

- 搜索速度快
- 匹配范围合理
- 排序符合用户直觉
- 搜索结果展示清晰

### 21.2 界面复杂化风险

如果主界面放太多新增、编辑、删除、平台管理入口，会变成后台管理工具。

应坚持：

- 主界面服务查找和复制
- 维护入口弱化
- 命令卡片只突出复制

### 21.3 数据维护成本风险

如果用户需要手动录入大量命令，初期使用门槛会较高。

后续可通过：

- 默认命令模板
- 导入功能
- AI 生成描述
- 从 Markdown 导入

降低维护成本。

### 21.4 平台分类过细风险

如果平台、分类、标签设计过多，用户可能不知道放在哪里。

第一版建议：

- 平台是一级组织
- 分类是平台内浏览
- 标签用于搜索
- 不做复杂多级目录

## 23. 当前结论

第一版产品应明确聚焦：

```text
选择平台 → 查看命令 → 搜索命令 → 一键复制
```

CRUD 是必要能力，但不是产品主路径。

推荐的第一版主界面是：

```text
左侧平台导航 + 顶部搜索 + 右侧命令卡片列表
```

推荐的第一版功能优先级是：

```text
搜索 > 平台选择 > 命令展示 > 一键复制 > 收藏/最近使用 > 低频维护
```

这会让产品更像一个真正高频使用的开发者效率工具，而不是一个命令数据管理后台。
